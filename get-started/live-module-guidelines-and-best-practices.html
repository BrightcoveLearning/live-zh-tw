    即時模組指南與最佳實務

    透過 Video Cloud Live，您可以輕鬆設定即時活動，並將多位元速率串流傳遞至網路、iOS 和 Android 裝置。本主題概述了一系列的最佳做法和建議，以協助確保高品質、穩定的直播體驗。
    注意：即時串流支援僅適用於視訊雲端專業版和企業發佈商。

    如果您想要快速開始直播直播活動，請查看：

    • 逐步：使用即時模組與電流線上傳送
    • 逐步：使用開放式廣播軟體 (OBS) 的直播模組
    有關Brightcove提供的用於傳輸實時流視頻的所有選項的列表，請參見提供實時流。

    認證編碼器

    有關受支持的編碼器的列表，請參見支持的直播活動編碼器。其他基於硬件和軟件的編碼工具可能適用於 Live 模塊，但不支持它們的使用。如果您對特定工具有疑問，請聯繫Brightcove支持欲獲得更多信息。

    輸入流指導方針

    提供高品質、穩定的輸入串流是確保觀眾獲得最佳使用者體驗的唯一途徑。良好的輸入串流以最高一致的可用頻寬提供最佳的視訊品質。

    輸入限制

    為了確保最高品質、最一致的串流體驗，Brightcove Live 將輸入串流設定限制為：

    • 最大值
    • 最多 30 fps，這是典型的。Brightcove最高支持60 fps，但您需要聯繫Brightcove支持增加限額。使用 60fps 時，Brightcove 建議大幅增加位元速率並使用固定的影格速率。
    • 低於 10 Mbps
    • 恆定位元率 (CBR) 可大幅降低問題的機會
    • 音訊取樣率：建議使用 44.1 kHz 和 48 kHz 的取樣音訊速率
    • 關鍵影格速率或 GOP (圖片群組) 對齊：
      1. 一種關鍵幀應該總是每2秒發生一次用於輸入和輸出。意思是每隔兩秒串流本身就會從編碼器傳送關鍵影格到 Brightcove。可以用不同的方式定義此過程，但是關鍵幀率是最常見的。
      2. 它可以通過不同的編碼器以不同的方式計算。例如：
        • Wirecast 會使用傳遞的影格數量，因此對於 30fps 的視訊，設定為 60。
        • 元素編碼器使用秒，以便正確的設置將是 '2'。
        • 60 FPS 影片只會在此設定被影格計算時變更，在這種情況下，每 120 個影格會等於 2 秒。
    • 如果有選擇關鍵幀對齊，同步GOP，對齊關鍵幀或類似的內容，請確保關鍵幀對齊。當關鍵影格未對齊時，會造成 HLS 分割的同步問題。

    測試頻寬

    達到輸入流的適當設置的第一步是確定現場的可用帶寬。有幾個工具可以幫助：

    • SpeedOf.Me（ http://speedof.me）-確定可用於HTTP連接的總帶寬是一個很好的第一步。不過，由於輸入摘要會透過 RTMP 而非 HTTP 串流至即時模組，因此 RTMP 連線的實際可用頻寬將會大幅減少。
    • 速測（ http://www.speedtest.net）-用於確定當前上傳和下載速度的在線工具。

    輸入頻寬

    提供高品質、穩定的輸入串流是確保觀眾獲得最佳使用者體驗的唯一途徑。良好的輸入串流以最高一致的可用頻寬提供最佳的視訊品質。

    • 最小輸入頻寬：2.5 兆
    • 最大輸入頻寬：10 兆
    注意：為獲得最佳的使用者體驗，請將輸出轉譯與編碼器串流的品質進行比對。

    確定編碼器功能

    了解用來編碼即時串流並將其傳送至 Live 模組的軟體和硬體功能也很重要。可能有大量的位元速率來傳送高品質的 1080p 輸入串流，但硬體也需要能夠以比即時速度更快的速度進行編碼。某些編碼工具會顯示有關所使用的總 CPU 使用率和頻寬的資訊。例如，電流 Wirecast 會在頁面頂端顯示輸出統計資料。此資訊在判斷指定硬體上可能的最穩定、最高品質的串流時非常有用。

    • 資料表應該靠近目標位元速率

    準備事件

    在使用即時模組進行生產活動之前，測試整個端對端鏈對確保成功至關重要。測試主要有兩個領域：

    • 視訊製作工作流程
    • 用來連線至即時模組的工具

    對於每個新的位置，或任何時候硬件或軟件配置更改，都應該執行新一輪測試。

    生產

    正在生成並發送到編碼硬件或軟件的即時饋送可以像網絡攝像頭一樣簡單，並且像多攝像頭一樣複雜，編輯的直播。在任何一種情況下，傳入的視訊摘要都必須在整個事件中保持一致。確保在攝影機角度或系統之間切換會產生相同的解析度和音訊頻道數量。

    連線至即時模組

    與 Live 模組的連線應該在相同的網路上測試，使用相同的輸入即時饋送，透過與生產事件所使用的相同編碼軟體傳送。目標是盡可能地模仿生產事件。

    活動當天

    您應該規劃比排定的廣播前 10 分鐘開始直播。提前 30 分鐘開始將提供一些額外的時間來協調和解決任何問題。

    即時串流期間的問題

    測試即時串流、尋找輸入串流的理想設定，以及為事件準備場地的工作，將大幅降低發生錯誤的機會。但是，始終準備好處理可能出現的任何問題是非常重要的。

    中斷連線

    網路中斷、編碼器故障或電源中斷可能導致 Live 模組連線中斷。現場直播有一個重新連接時間，這是等待流重新連接的時間，以分鐘為單位。當連線中斷時，Live 模組會在選擇的「重新連線時間」之後中斷連線。通過展開重新設置時間高級選項創建事件時：

    在此重新連線期間，Live 模組將接受任意數量的中斷連線並重新連線，而不需要開始新的串流。串流重新連接後，將從最近的影格繼續播放。如果事件結束而沒有重新連線串流，玩家將會傳回警告，指出找不到串流。

    注意：中斷連線並重新連線時，串流設定必須保持不變。對音訊頻道數目、解析度或轉碼器設定所做的任何變更，都會導致無法預測的行為。

    延遲

    串流延遲，或觀眾正在觀看的事件與實際即時事件之間的時間差異，分為三個地方介紹：

    • 現場編碼器
    • 雲端轉碼
    • CDN 交付

    現場編碼器和 Live 模組的雲端轉碼系統總共帶來 1.5 至 5 秒的延遲。根據 CDN，可能會引入另一個 15-90 秒的延遲。一般而言，即時動態消息會被視為 15-90 秒之間的延遲是正常的，而 Live 模組必須能夠為數百萬觀眾觀看直播活動的同時觀眾提供服務。

    即時串流問題疑難排解

    問題 解析度
    播放凍結，音訊繼續 檢查編碼器的輸入摘要，並檢查編碼器是否仍然連接正確，並對視頻進行編碼。如果有中斷連線，請確定串流設定相同，然後再重新連線。
    播放器中的串流顯示凍結 Live 模組很可能停止接收來自輸入編碼器的資料。確認輸入串流處於作用中狀態、已連線，並且傳送資料速度比即時更快。
    不穩定的播放 確定有足夠的頻寬可支援輸入串流的位元速率。斷續播放通常表示生產端出現錯誤或頻寬不足的指示。
    空白螢幕 確認有足夠的頻寬。如果中斷連線，請確定在重新連線之前未變更任何設定。
    超過預期的延遲 檢查可用頻寬，並確保資料是否即時傳送至即時模組。串流可能需要 10 分鐘才能發佈。在此期間，會快取資料流，而延遲將會是最長的。發佈後，延遲會持續減少，直到穩定在 30-90 秒之間。
    已根據您的帳戶用盡您分配的冗餘串流。 您可能會有多餘的群組計算您的帳戶限制。一旦具有冗餘流的事件完成或被取消，它就不再可見在即時模塊中。不過，這類事件可以透過即時 API 列出和刪除。請參閱實時API：備援群組。
     

    如需進一步說明

    如果您需要進一步協助讓您的直播活動正常運作，請與我們聯絡。為了確保您能夠獲得最快速的回應，以下是 Brightcove 支援部門解決問題所需的清單。

    • 流的具體症狀。例如，它是根本不播放還是口吃或凍結？
    • 這個流是否在過去正常工作
    • 您在編碼器中使用的入口點 URL
    • 您使用的編碼軟件和硬件
    • 您發佈直播活動的玩家 URL
    • 影片雲端工作室中即時資產的視訊 ID
    • 從編碼器到發佈點主機的追蹤路由結果
